# Method: Four-Domain Retrieval-Augmented Trial Profiling

## Overview

We construct a structured retrieval-augmented generation (RAG) profile for each clinical trial by integrating evidence from four complementary biomedical sources: disease knowledge from PrimeKG, drug annotations from DrugBank, biomedical literature from PubMed, and similar interventional studies from ClinicalTrials.gov. The pipeline is designed to transform a raw ClinicalTrials.gov protocol record into a compact, machine-readable profile containing four fields: `disease_info`, `drug_info`, `relevant_papers`, and `similar_trials`. These fields provide disease-level background, intervention-level pharmacological evidence, literature support, and trial-design precedents, respectively. The resulting profile is intended to serve as a standardized evidence context for downstream eligibility-criteria generation, clinical trial comparison, or benchmark evaluation.

For benchmark construction, the input is a JSON array of ClinicalTrials.gov protocol records. For each record, we extract the NCT identifier, official title, intervention list, primary outcomes, and eligibility criteria from the `protocolSection`. The collector processes records independently and writes one JSON file per trial, while maintaining a manifest keyed by NCT identifier to support resumable execution. To avoid conflating retrieval failures with missing evidence, each retrieval component returns the string `none` when no usable context is available; otherwise, the component emits a formatted text block that preserves source identifiers such as PMIDs and NCT IDs.

## Trial Parsing and Query Construction

Each protocol record is first converted into a normalized internal representation. Drug candidates are extracted from the `armsInterventionsModule` by retaining interventions whose type is `DRUG`. Primary outcomes are serialized from the `outcomesModule`, and eligibility criteria are copied from the `eligibilityModule` when available. Because the benchmark records do not always expose a clean disease-condition field, the system infers the disease query from both the official title and eligibility criteria. A rule-based extractor searches for high-precision textual patterns such as "patients with", "treatment of", and "diagnosed with"; it also checks a curated list of frequent disease phrases that occur in the benchmark set. The extracted disease phrase is then passed to a language model for medical normalization, which maps noisy phrases, abbreviations, and compound descriptions to a single standardized disease term.

The language model is used only for term normalization and candidate disambiguation, not for generating the final evidence fields. In the current implementation, the model is accessed through an Anthropic-compatible MiniMax endpoint and is prompted to return valid JSON. For disease normalization, the model returns a best disease term, a short definition, and key clinical characteristics. If the model call fails or returns malformed output, the pipeline falls back to the first extracted disease phrase, preserving batch progress while marking the corresponding evidence quality through the absence of downstream matches.

## Disease Knowledge Retrieval

The disease module retrieves structured disease information from a local PrimeKG-derived feature table [CITATION NEEDED: PrimeKG]. After disease normalization, the system filters candidate disease names using case-insensitive whole-word matching and suffix-insensitive matching over common disease descriptors such as "disease", "syndrome", "disorder", and "finding". When multiple candidate disease nodes remain, a two-stage language-model disambiguation procedure is applied. The first stage asks for a definition and key characteristics of the normalized disease term; the second stage selects the best candidate disease name from the retrieved candidates using the generated clinical analysis as context. The final disease entry aggregates all non-empty feature columns for the selected disease node, excluding metadata identifiers and internal grouping columns. This yields a text block containing definitions, clinical descriptions, risk factors, complications, and other structured disease attributes when available.

This design separates lexical recall from semantic precision. The candidate filter is deliberately permissive to avoid missing relevant disease nodes, while the disambiguation step resolves closely related disease variants, aliases, and broader or narrower disease entities. The final profile records the selected disease name explicitly so downstream components can distinguish the original trial wording from the normalized knowledge-base entity.

## Drug Knowledge Retrieval

The drug module retrieves intervention evidence from a processed DrugBank table [CITATION NEEDED: DrugBank]. Each extracted drug name is matched against canonical product names and, when enabled, synonym lists. The matcher preprocesses the DrugBank table once at initialization, builds lookup structures for product names and synonyms, and then searches each trial intervention independently. For every successful match, the profile includes pharmacologically relevant fields: canonical name, SMILES string, molecular formula, indication, mechanism of action, pharmacodynamics, absorption, metabolism, half-life, protein binding, route of elimination, volume of distribution, clearance, and toxicity. Empty, missing, and non-informative values are removed before formatting.

Drug retrieval is intentionally conservative: a trial may list non-drug interventions, placebo arms, dosage annotations, or compound descriptions that do not correspond to a DrugBank entity. The formatter therefore emits evidence only for successful matches and omits unmatched interventions rather than synthesizing pharmacological descriptions. When multiple trial drugs are matched, their DrugBank evidence blocks are concatenated in the order in which the interventions appear in the protocol record.

## PubMed Literature Retrieval and Reranking

The literature module retrieves disease- and intervention-specific biomedical papers from PubMed through NCBI Entrez. For a trial with normalized disease term \(d\) and extracted drug terms \(\{a_i\}\), the primary Boolean query is constructed as \((d) \land ((a_1) \lor \cdots \lor (a_n))\). If no intervention terms are available, the disease term alone is used. The system first queries PubMed for the total number of matches, fetches up to a configured maximum number of records, and retains articles with abstracts. Records are fetched in batches using Entrez `epost` and `efetch`, with retry handling for transient network errors. The default benchmark configuration retrieves up to 300 PubMed records per trial, with batch size 100 and two worker threads for detail fetching.

Retrieved PubMed articles are then converted into temporary documents whose content is the concatenation of title and abstract. The system embeds these documents using a multilingual MiniLM sentence embedding model and stores them in a local FAISS index [CITATION NEEDED: FAISS; CITATION NEEDED: sentence-transformers]. The trial title is used as the semantic query. The FAISS index first returns a candidate set, after which a BGE cross-encoder reranker scores query-document pairs [CITATION NEEDED: BGE reranker]. The final paper set consists of the top-ranked documents whose reranker score exceeds the configured similarity threshold. The benchmark configuration uses `BAAI/bge-reranker-large` and a paper similarity threshold of 0.5. The temporary FAISS index is deleted after reranking so that the persistent output contains only the selected article metadata and abstracts.

This two-stage retrieval procedure balances recall and precision. The Boolean PubMed query anchors retrieval to disease and intervention terminology, the dense FAISS search identifies semantically similar abstracts among the retrieved set, and the cross-encoder reranker applies a more expressive relevance model before evidence is written to the profile. Each selected paper is represented by PMID, title, and abstract, allowing downstream users to audit the source evidence directly.

## Similar Trial Retrieval

The similar-trial module retrieves design precedents from ClinicalTrials.gov separately for Phase 1, Phase 2, and Phase 3 studies. Before profiling, each phase-specific retrieval base is prepared as a CSV table containing NCT ID, title, description, disease, keyword, eligibility criteria, intervention name, and outcome measure. Each trial in the retrieval base is also encoded with a pretrained Trial2Vec model [CITATION NEEDED: Trial2Vec], producing phase-specific embedding dictionaries stored as `Phase1_emb_only_fields.pth`, `Phase2_emb_only_fields.pth`, and `Phase3_emb_only_fields.pth`. At runtime, these embeddings are loaded together with the corresponding retrieval-base CSV files; retrieval is disabled if either component is missing.

For each query trial, the system constructs a synthetic one-row Trial2Vec input containing the trial title, concatenated drug names, normalized disease term, and keywords. The pretrained Trial2Vec encoder maps this query into the same embedding space as the phase-specific retrieval bases. For each phase, the system L2-normalizes the query embedding and all candidate trial embeddings, computes cosine similarity by dot product, and retrieves the top candidate trials. Candidates are then filtered by three criteria: similarity must exceed the configured threshold, the candidate disease field must share at least one normalized word with the query disease term, and the candidate title must not match the query title after punctuation and whitespace normalization. The surviving candidates are sorted by similarity, and up to five trials per phase are written to the profile.

The final `similar_trials` field groups the retrieved trials under phase-specific headers. Each trial entry includes the NCT ID and available metadata fields from the retrieval base, including title, description, disease, keywords, eligibility criteria, intervention names, and outcome measures. This phase-aware retrieval design provides downstream models with examples from the same broad stage of clinical development while preserving visibility into cross-phase differences in eligibility criteria, intervention formulation, and outcome selection.

## Output Representation

After all four retrieval modules have completed, the internal result is converted into a compact profile schema. `Disease_info` is mapped to `disease_info`, `Drugbank_info` to `drug_info`, `Papers_info` to `relevant_papers`, and the phase-specific clinical-trial fields are concatenated into `similar_trials`. If no similar trial is retrieved for any phase, `similar_trials` is set to `none`; otherwise, phase sections are separated by explicit headers such as `--- Similar Phase3 Trials ---`. The output is serialized as UTF-8 JSON with non-ASCII characters preserved, which is important for biomedical symbols, drug names, and registered-trademark characters.

The collector writes one profile per trial using a title-derived filename and resolves name collisions with progressively shorter title slugs or the NCT ID. A manifest file records the output filename and title for each processed NCT ID, enabling interrupted runs to resume without recomputing completed trials. A `--force` option regenerates existing profiles when retrieval data or model configuration changes. This mechanism is important for benchmark reproducibility because the RAG profile is sensitive to the availability of external retrieval bases, reranker thresholds, and the disease-normalization model.

## Reproducibility Details

All paths are specified relative to the project root through a JSON configuration file. The default embedding model for PubMed abstracts is `paraphrase-multilingual-MiniLM-L12-v2`; the default reranker is `BAAI/bge-reranker-large`; PubMed retrieval uses an NCBI Entrez email but no PubMed API key; and similar-trial retrieval uses phase-specific Trial2Vec embeddings with a default cosine-similarity threshold of 0.5. Disease evidence is read from `data/primekg/disease_features.csv`, drug evidence from `data/drugbank/drugbank_data_v1.csv`, and similar-trial retrieval from paired files under `data/trial_gov/embeddings/` and `data/trial_gov/retrieval_base/`. For benchmark collection, the command-line entry point processes `final_bench.json` and emits profiles to `outputs/bench_profiles/`.

The resulting method is deterministic after disease normalization and candidate disambiguation have completed. Remaining nondeterminism arises primarily from external API availability, PubMed database updates, and the language-model endpoint used for normalization. To support auditability, the generated profile preserves source identifiers for all retrieved literature and clinical trials, and missing evidence is represented explicitly rather than filled by generative text.
